## 2. 진단 — 4가지 구조적 부재

### A. 가시성 부재 (Visibility) — #1, #7, #10
업무 현황이 각자의 머릿속과 카톡에만 존재.  
→ **내 업무 홈** + **팀 전체 현황 뷰**

### B. 소통 채널 분산 (Communication) — #2, #3
업무 요청과 일상 대화가 카톡 하나에 섞임 → 맥락 소멸.  
→ **프로젝트 카드에 요청자·맥락 고정** + 빠른 업무 추가 (3필드 입력)

### C. 프로젝트 구조 부재 (Structure) — #4, #5, #6
"프로젝트"라는 그릇 자체가 없음. 자료·사람·업무가 흩어짐.  
→ **프로젝트 허브** (칸반 + 문서함 + 마일스톤)

### D. 책임 체계 부재 (Accountability) — #8, #9
역할 미정의 + SOP 미연결.  
→ **담당자/확인자/요청자 필수 지정** + SOP 연결

## 3. 핵심 전략

```
[기존]  카톡 지시 → 각자 처리 → 결과 카톡 → 기록 소멸
[신규]  프로젝트 생성 → 항목 추가(담당자/마감) → 문서 연결 → 완료 체크 → 자동 기록
```

> 카톡 대체 성공 조건: **카톡보다 빠른 업무 추가 UX** (제목+담당자+마감 = 3필드)

### 캘린더 내 루틴 등록 스펙

개인이 반복 업무를 캘린더에 직접 등록하는 기능.

```
루틴 등록 필드:
  제목         ex) "매장 재고 확인"
  주기         매일 / 매주 (요일 선택) / 매월 (일 선택)
  시간대       오전 / 오후 / 종일
  알림         시작 N분 전
  부서/개인     개인 루틴 vs 팀 루틴

캘린더 표시:
  루틴 이벤트는 반복 아이콘(↺)으로 구분
  해당 날짜에 완료 체크 가능
  미완료 시 다음날 이월 여부 설정 가능
```

Firestore: `salesDb > calendar_routines/{routineId}`
```
id, title, ownerId, ownerName
recurrence: 'daily' | 'weekly' | 'monthly'
weekdays?: number[]    // 매주: 0=일~6=토
monthDay?: number      // 매월: 1~31
timeSlot: 'morning' | 'afternoon' | 'allday'
notifyMinutes?: number
isTeamRoutine: boolean
departmentId?: string
isActive: boolean
createdAt
```

루틴 완료 기록: `salesDb > routine_completions/{date_routineId}`
```
routineId, ownerId, date (YYYY-MM-DD), completedAt
```

## 5. 프로젝트 기능 상세 스펙

### 5-1. 목록 뷰 (상단 탭)

```
[진행중] [전체] [시각화]      [🔍 검색]  [+ 새 프로젝트]
```

- **진행중**: status = 'active' 프로젝트만, 마감 임박 순 정렬
- **전체**: 완료·아카이브 포함 전체 목록
- **시각화**: 타임라인 or 진행률 카드 그리드

### 5-2. 프로젝트 속성

```
제목, 설명
상태: 활성 / 보류 / 완료 / 아카이브
오너(총괄), 멤버 목록
시작일, 마감일
마일스톤: [{ 제목, 날짜, 완료여부 }]
브랜드 태그 (전사통합이지만 연관 브랜드 태깅 가능)
```

### 5-3. 프로젝트 내부 구성

```
프로젝트 상세 페이지
├── 개요 (제목, 설명, 마일스톤 진행률 바)
├── 칸반 보드
│   ├── 할 일 컬럼
│   ├── 진행 중 컬럼
│   └── 완료 컬럼
│       └── 카드: 제목 + 담당자 + 확인자 + 요청자 + 마감일 + 우선순위
├── 문서함
│   ├── 프로젝트 전용 문서 (새로 작성)
│   └── 기존 문서 연결 (보고서 / 회의록 / 일일보고 / 주간보고)
└── 활동 로그
```

### 5-4. 칸반 카드 필드

| 필드 | 필수 | 설명 |
|------|------|------|
| 제목 | ✅ | |
| 담당자 (Responsible) | ✅ | 실행하는 사람 |
| 확인자 (Accountable) | 권장 | 완료 검수·승인 |
| 요청자 (Requester) | 권장 | + 요청 맥락 메모 |
| 마감일 | 권장 | D-1에 담당자·관리자 알림 |
| 우선순위 | ✅ | 긴급/높음/보통/낮음 (색상 태그) |
| SOP 연결 | 선택 | 관련 업무 규정 연결 |

### 5-5. 첫 번째 템플릿 — 신메뉴 개발

```
신메뉴 개발 프로젝트 [템플릿]
├── [할 일] 컨셉 기획서 작성
├── [할 일] 식재료 원가 산출
├── [할 일] 시제품 제조
├── [할 일] 내부 시식 평가
├── [할 일] 원가율 최종 확인
├── [진행 중] → 해당 항목 이동
└── [완료] → 완료 항목

마일스톤:
- 컨셉 확정
- 시제품 완성
- 원가 승인
- 출시 확정
```

## 6. Firestore 구조 (salesDb)
